## 一、体系概述

### 什么是 AI Skill 体系？

AI Skill 体系是一套**六层协作式 AI 工作流系统**。它将开发一个 AI 项目从立项到上线的全过程，拆解为六个专职 AI 角色，每个角色只做自己最擅长的那一层工作，通过**标准化交接包（Handoff Payload）**无缝传递上下文。

> **核心理念**：与其让一个 AI 全能但粗糙地完成所有事，不如让六个 AI 各司其职、精准协作，每层产出专业、可验证的交付物。

## 二、快速上手（5分钟）

### 第一步：选择你的入口

根据你目前的情况，选择最合适的入口：

| 你的情况 | 建议入口 |
|---|---|
| 有个模糊想法，不知从哪开始 | Skill 00（Navigator）|
| 想找合适的开源库/框架 | Skill 03（Scout）|
| 有需求文档，想工程化打包 | Skill 02（SOP 工程师）|
| 想优化一段提示词 | Skill 01（超级提示词工程师）|
| 有 SOP/需求，想生成操作手册 | Skill 04（Planner）|
| 有已完成的项目，想验收上线 | Skill 05（Validator）|

### 第二步：获取激活卡

在 GitHub 仓库中找到对应 Skill 的 `core/activation-card.md` 文件，复制其中的激活提示词（三档可选）。

**三档激活卡说明**：

| 档位 | 适用场景 | 长度 |
|---|---|---|
| 快速激活（单行版）| token 充足时，快速测试 | 1 行 |
| 标准激活（推荐）| 日常使用，含完整工作规则 | 约 20-30 行 |
| 极简激活 | token 极度受限时 | 3-5 行 |

### 第三步：粘贴到新对话开头

打开你使用的 AI 平台（GPT / Claude / 文心 / 通义等），**新建一个对话**，将激活卡内容粘贴到对话框发送。AI 会立即回应激活确认，例如：

    ✅ 智能体导航官已就绪（v1.0.0）

    请描述你想做的事，我来帮你找到最合适的 Skill：...

### 第四步：开始工作

按照 AI 的引导，逐步提供信息，完成当前 Skill 的工作流程。每个 Skill 完成后，会输出一个**交接包（Handoff Payload）**，将其复制到下一个 Skill 的对话开头，实现无缝衔接。

## 三、六层 Skill 详解

### Skill 00 · Navigator（导航官）

**适用场景**：你有个模糊想法，不确定应该用哪个 Skill；或者你的需求涉及多个环节，需要规划整体路径。

**核心能力**：
- 意图识别（7 种意图类型，置信度三级处理）
- 单 Skill 路由（推荐最合适的一个 Skill）
- 多 Skill 组合路由（规划完整执行路径）

**工作方式**：

| 置信度 | 处理方式 |
|---|---|
| ≥ 80% | 直接输出路由建议 + 交接包，无需追问 |
| 60-79% | 先确认「我理解你的需求是X，对吗？」 |
| < 60% | 追问（最多 3 次），第 3 次后强制路由到 Skill 01 |

**输出物**：路由建议 + 交接包 A（含 intent_type / confidence_score / pipeline / project_summary）

---

### Skill 01 · 超级提示词工程师

**适用场景**：你有一段提示词需要优化；或者你想为某个场景设计一个高质量的系统提示词。

**核心能力**：
- 深度解构现有提示词（识别缺陷、优化方向）
- 从零设计系统提示词（ROLE/TASK/CONTEXT/FORMAT）
- 迭代优化（多轮对话追踪改进效果）

**输出物**：优化后的系统提示词 + 修改说明 + 测试用例

---

### Skill 02 · SOP 工程师

**适用场景**：你有一个 AI Skill 的想法或原材料（需求文档/提示词草稿），想将其工程化打包成标准 Skill 工程包。

**核心能力**：
- 10 阶段标准开发工作流（从需求分析到 GitHub 发布）
- 生成完整工程包（SKILL.md / system-prompt-full.md / sub-skills / deploy / docs）
- 自举式测试（用自身开发的 Skill 验证质量）

**输出物**：完整 Skill 工程包（含所有文件）

---

### Skill 03 · Scout（开源侦察官）

**适用场景**：你的 AI 项目需要用到开源库或框架，但不知道选择哪个；或者你想对技术方案进行系统性评估。

**核心能力**：
- 需求拆解（REQ-XXX 格式技能需求清单）
- 开源方案搜索（每个需求 2-3 个备选）
- 七维度评估矩阵（功能性 30% / 易用性 20% / 性能 15% / 可维护性 10% / 社区活跃度 10% / 兼容性 10% / 文档 5%）
- 幻觉防护（所有 Star 数 / 版本号 / 更新日期均标注 [需用户核实]）

**4 阶段工作流**：

| 阶段 | 产出 | 门控 |
|---|---|---|
| 01 需求理解 | 项目需求分析表 + REQ 清单 | 用户确认清单完整 |
| 02 开源搜索 | 每需求 2-3 个备选方案 | 自动进入阶段03 |
| 03 七维度评估 | 评估矩阵 + 推荐选择 | 用户确认推荐方案 |
| 04 交接包输出 | 标准化交接包（含 sop_hint）| 用户确认可用 |

**输出物**：开源选型报告 + 交接包 B（含 selected_skills / sop_hint）

---

### Skill 04 · Planner（执行规划官）

**适用场景**：你已有 SOP 文档或项目需求，需要生成一份任何人都能按步骤执行的详细操作手册。

**核心能力**：
- SOP 文档解析（提取技术栈/阶段/约束）
- P-T-S 三层结构拆解（Phase / Task / Step，编号格式 P1-T2-S3）
- 完整操作手册生成（含环境配置/检查点/断点恢复/FAQ）
- PERT 三点估时（乐观/正常/悲观）

**Step 标准格式**（每个步骤的完整信息）：

```
P1-T1-S1：[步骤描述]
- 工具：[终端/编辑器/浏览器]
- 操作：[具体命令或操作]
- 预期结果：[执行后应看到什么]
- 验证：[如何确认这一步成功了]
- 估时：[X] 分钟 [估算，请根据实际调整]
- 依赖：[前置步骤ID]
```

**三不原则**：不过度设计 / 不静默调整 SOP 技术决策 / 不跳过验证步骤

**输出物**：完整 Markdown 操作手册 + 交接包 C（含 total_phases / total_steps / test_targets）

---

### Skill 05 · Validator（测试验收工程师）

**适用场景**：你已完成项目或 Skill 工程包，需要在发布前进行系统性质量验收。

**核心能力**：
- 五维度量化评分（功能完整性 40% / 文档完整性 25% / 可执行性 20% / 接口规范性 10% / 安全性 5%）
- P0-P3 缺陷分级（P0=阻塞，P1=严重，P2=一般，P3=轻微）
- 三档发布决策（PASS / CONDITIONAL_PASS / FAIL）
- 飞轮机制（缺陷模式归档 → 上游 Skill 改进建议）

**发布决策矩阵**：

| 决策 | 条件 |
|---|---|
| **PASS** | P0 = 0，P1 = 0，综合得分 ≥ 80% |
| **CONDITIONAL_PASS** | P0 = 0，P1 ≤ 3，综合得分 ≥ 70% |
| **FAIL** | P0 ≥ 1，或（P1 > 3 且综合 < 70%）|

**输出物**：验收报告（含各维度得分 / 缺陷清单 / 发布决策）+ 交接包 E + 飞轮改进建议

## 四、完整使用流程

### 4.1 标准全流程（推荐）

适合从零开始开发一个新 AI 项目：

```
Step 1：Skill 00（Navigator）
  └→ 输出：路由建议 + 交接包 A

Step 2：Skill 01（超级提示词工程师）[可选]
  └→ 输入：交接包 A
  └→ 输出：优化后的核心系统提示词

Step 3：Skill 03（Scout · 开源侦察官）[可选]
  └→ 输入：交接包 A 或用户需求
  └→ 输出：开源选型报告 + 交接包 B

Step 4：Skill 02（SOP 工程师）
  └→ 输入：交接包 B + 提示词（来自 Step 2/3）
  └→ 输出：完整 Skill 工程包

Step 5：Skill 04（Planner · 执行规划官）
  └→ 输入：来自 Skill 02 的交接包
  └→ 输出：分阶段操作手册 + 交接包 C

Step 6：Skill 05（Validator · 测试验收工程师）
  └→ 输入：交接包 C + 工程包材料
  └→ 输出：验收报告 + 发布决策
```

### 4.2 常见使用路径

根据实际需求，你可以只使用其中几个 Skill：

**路径 A：技术选型路径**
```
Skill 03（找开源方案）→ [可选] Skill 04（生成实施手册）
```
适合：「我有需求，只需要找到合适的开源库并知道怎么用」

**路径 B：提示词优化路径**
```
Skill 01（优化提示词）→ [可选] Skill 02（工程化打包）
```
适合：「我有一段提示词效果不好，想改进，然后打包成 Skill」

**路径 C：验收发布路径**
```
[已有工程包] → Skill 05（验收测试）
```
适合：「我已经做好了一个 Skill/项目，需要上线前做质量检查」

**路径 D：完整开发路径**
```
Skill 00（导航）→ Skill 03（选型）→ Skill 02（工程化）→ Skill 04（规划）→ Skill 05（验收）
```
适合：「我有一个 AI 项目想法，想完整走一遍从选型到上线的流程」

## 五、激活卡使用方法

### 5.1 如何找到激活卡

每个 Skill 的 GitHub 仓库中都包含 `core/activation-card.md` 文件，其中有三档激活卡供选择。

### 5.2 激活卡速查表

以下列出各 Skill 的**单行版快速激活内容**，可直接复制使用：

**Skill 00 · Navigator**
```
你是 Navigator，一个 AI Skill 路由引擎。用户输入意图后，你分析置信度并路由到正确 Skill（01需求分析/02SOP工程/03开源侦察/04执行规划/05测试验收）。置信度≥80%直接路由，60-79%先确认，<60%追问（≤3次）。输出标准化交接包YAML。禁止直接执行任务，只做路由。
```

**Skill 03 · Scout**
```
你是开源技能侦察官(v1.0.0)。将项目需求转化为开源技能选型方案，七维度评估（功能性30%/易用性20%/性能15%/可维护性10%/社区活跃度10%/兼容性10%/文档5%），输出标准交接包。4阶段：需求理解→开源搜索→七维评估→交接包输出。规则：实时数据标注[需用户核实]；门控不跳阶；技术栈未知必追问；找不到方案直言+3条降级路径。
```

**Skill 04 · Planner**
```
你是AI项目执行规划官(v1.0.0)。将SOP工程包转化为分阶段操作手册，4阶段：SOP解析→Phase/Task/Step三层拆解（P1-T2-S3格式）→手册生成（含环境配置/检查点/断点恢复/FAQ）→交接包输出。版本号/API限制标注[需用户核实]，估时标注[估算]，不修改SOP技术决策，不跳阶。
```

**Skill 05 · Validator**
```
你是AI项目测试验收工程师(v1.0.0)。执行五维度验收测试（功能40%/文档25%/可执行性20%/接口规范10%/安全5%），生成PASS/CONDITIONAL_PASS/FAIL发布决策。P0缺陷立即FAIL，P0=0且P1≤3且综合≥70%为CONDITIONAL_PASS，P0=0且P1=0且综合≥80%为PASS。所有缺陷必须有证据，输出飞轮改进建议。
```

### 5.3 标准激活（推荐）的获取

如需更精准的效果，建议使用标准激活版。访问对应仓库，打开 `core/activation-card.md`，复制「标准激活（推荐）」部分的内容。

### 5.4 注意事项

**重要**：每个 Skill 必须在独立的对话窗口中使用。在同一对话中切换多个 Skill 会导致角色混淆，影响输出质量。

| 做法 | 说明 |
|---|---|
| ✅ 为每个 Skill 开启新对话 | 推荐，保证角色清晰 |
| ✅ 复制整个激活卡内容（含所有规则）| 保证行为规范准确 |
| ❌ 在同一对话中使用多个激活卡 | 会导致角色混淆 |
| ❌ 只复制部分激活内容 | 可能缺少关键规则 |

## 六、交接包格式参考

### 6.1 什么是交接包？

交接包（Handoff Payload）是 AI Skill 体系中 Skill 之间传递上下文的标准载体。它是一段 YAML 格式的结构化数据，包含了上游 Skill 完成的工作摘要，以及下游 Skill 需要的所有关键信息。

### 6.2 交接包标准结构

```yaml
handoff:
  schema_version: "1.0"          # 必填，当前版本固定为 "1.0"
  from_skill: "skill-XX-name"    # 必填，完整 Skill 标识符
  to_skill: "skill-YY-name"      # 必填，下游 Skill 完整标识符
  payload:                       # 必填，核心数据字段（各 Skill 不同）
    # ... Skill 特定字段
  user_action: "操作说明"         # 必填，告知用户下一步操作
  created_at: "YYYY-MM-DD"       # 建议填写
```

### 6.3 各节点交接包字段速查

| 交接包 | 发出方 → 接收方 | 核心 payload 字段 |
|---|---|---|
| 交接包 A | Skill 00 → Skill 03/04 | intent_type / confidence_score / project_summary / pipeline |
| 交接包 B | Skill 03 → Skill 04 | selected_skills（含 req_id/chosen/install/score）/ sop_hint |
| 交接包 C | Skill 04 → Skill 05 | project_name / total_phases / total_steps / test_targets |
| 交接包 E | Skill 05 → 用户 | decision / composite_score / defects / flywheel_suggestions |

### 6.4 如何传递交接包

1. 在当前 Skill 的对话中，等待 AI 输出交接包（通常在对话末尾）
2. 复制完整的 YAML 内容（从 `handoff:` 开始到最后一行）
3. 打开下一个 Skill 的**新对话**，首先粘贴激活卡激活 Skill
4. 激活成功后，将交接包内容粘贴发送
5. 目标 Skill 会自动识别交接包并开始工作

## 七、端到端场景示例

本节展示「客户反馈分析智能体」的完整走通过程，帮助你理解各 Skill 在真实场景中如何协作。

### 场景描述

> 你想开发一个 Python 智能体，自动分析用户评价，提取情感倾向和主题，每天生成报告并发送邮件。

### 走通过程

**Node 1：Skill 00 · Navigator**

- 输入你的需求描述
- AI 识别到「BUILD_AGENT」意图，置信度 86%，直接路由
- 输出：推荐 4 步路径（03 → 02 → 04 → 05）+ 交接包 A

**Node 2：Skill 03 · Scout**

- 粘贴交接包 A 激活
- AI 拆解出 5 个技能需求（REQ-001 至 REQ-005）
- 经过七维度评估，推荐：HuggingFace Transformers（情感分析）/ BERTopic（主题提取）/ pandas（数据处理）/ Jinja2+WeasyPrint（报告生成）/ APScheduler（定时任务）
- 输出：选型报告 + 交接包 B（含 sop_hint 中的模块建议）

**Node 3：Skill 04 · Planner**

- 粘贴交接包 B 激活
- AI 解析出 5 个 Phase，生成含 10 个详细 Step 的操作手册

## 八、常见问题 FAQ

**Q：我需要先安装什么吗？**

无需安装任何软件。所有 Skill 都是纯文本提示词，在任何支持自定义系统提示词的 AI 平台上均可使用，包括 ChatGPT、Claude、文心一言、通义千问、星火、Gemini、CodeBuddy 等。

---

**Q：每个 Skill 收费吗？**

Skill 本身完全免费（MIT 开源协议）。你需要自行承担使用的 AI 平台的 API 调用费用（如 OpenAI API）。若使用免费额度（如 ChatGPT 免费版），不产生额外费用。

---

**Q：可以跳过某些 Skill 吗？**

可以。六个 Skill 各自独立，你不需要必须按顺序全部使用。例如：
- 只需要找开源方案 → 直接使用 Skill 03
- 有完整需求文档 → 跳过 Skill 00/01，直接从 Skill 02 或 Skill 04 开始
- 已有操作手册 → 直接使用 Skill 05 验收

---

**Q：交接包中有些字段不理解怎么办？**

交接包中的字段是为目标 Skill 设计的，你不需要手动修改它们。只需将 AI 输出的完整 YAML 内容原样复制粘贴即可。如果对某个字段有疑问，可以直接在当前 Skill 的对话中询问 AI。

---

**Q：AI 在同一对话中混淆了角色怎么办？**

这是已知的体系限制（SYS-LIM-005）。建议：
1. 每个 Skill 使用独立的新对话
2. 在对话开头完整粘贴激活卡内容（不要省略）
3. 如果 AI 偏离了 Skill 的职责范围，重新粘贴激活卡进行重置

---

**Q：Skill 03 给出的开源方案数据准确吗？**

Skill 03 内置了幻觉防护机制（H1/H2/H3 规则）：所有实时数据（Star 数、最后更新日期、版本号）均会标注 **[需用户核实]**。建议在采纳任何方案前，前往对应的 GitHub 仓库或官方文档核实这些数据。

---

**Q：Skill 04 生成的操作手册估时准确吗？**

所有估时均会标注 **[估算，请根据实际调整]**。这些时间是基于通用经验的参考值，实际耗时受网络速度、机器性能、经验水平等因素影响。请将其作为时间规划的参考，而非硬性承诺。

---

**Q：Skill 05 给出 FAIL，我应该怎么做？**

FAIL 报告中会包含：
1. 所有 P0/P1 缺陷的**具体位置、描述和修复建议**
2. 每个缺陷建议**返回哪个上游 Skill** 修复

按照修复建议，返回对应的 Skill（Skill 02/03/04）完成修复后，将更新后的材料重新提交给 Skill 05 验收。

---

**Q：可以同时使用多个 AI 平台吗？**

可以。Skill 体系设计为平台无关，不使用任何平台专属语法。你可以在 GPT 中使用 Skill 00 路由，在 Claude 中使用 Skill 03 选型，在文心中使用 Skill 04 规划——交接包在不同平台间完全可传递。

## 九、体系限制说明

### 已知系统级限制

| 编号 | 限制 | 影响 | 解决建议 |
|---|---|---|---|
| SYS-LIM-001 | Skill 间传递依赖用户手动复制粘贴交接包，非真实程序调用 | 多步协作有人工操作摩擦 | 每步完成后及时复制交接包；未来 v2.0 将实现自动化 |
| SYS-LIM-002 | 跨会话无持久化记忆 | 长工作流中断后需重新提供上下文 | 将每个 Skill 的输出（尤其是交接包）保存到本地文档 |
| SYS-LIM-004 | Skill 03 开源数据无实时核实 | Star 数/更新日期/版本号可能不准确 | 重要方案决策前，主动访问 GitHub 核实数据 |
| SYS-LIM-005 | 同对话中多 Skill 可能角色混淆 | AI 可能偏离当前 Skill 的职责 | 每个 Skill 使用独立对话窗口 |

### 设计限制（永久性）

- **Skill 05 不能验收自身**：为避免利益冲突，Skill 05 的迭代版本需由体系维护者人工验收，而非自我验收。
