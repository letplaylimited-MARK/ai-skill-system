## ADR-001：六层架构设计（为何不做成统一的单体 AI）

**状态**：已采纳 · v1.0.0 确立

**决策时间**：体系初始设计阶段

### 背景

在设计体系时，存在两种根本不同的方向：
1. **单体 AI 方案**：训练或提示工程一个能做所有事的超级 AI
2. **分层 Skill 体系**：多个专注单一职责的 Skill，通过标准化接口协作

### 分析

| 维度 | 单体方案 | 六层 Skill 体系 |
|---|---|---|
| 上下文压力 | 单次对话承载所有能力规则，Context 消耗大 | 每个 Skill 独立对话，Context 聚焦单一职责 |
| 质量深度 | 每种能力浅尝辄止，互相干扰 | 每个 Skill 可深度定制，输出质量高 |
| 可维护性 | 修改一处可能影响全局 | 单 Skill 升级不影响其他 Skill |
| 可测试性 | 难以隔离测试单个能力 | 每个 Skill 可独立验收（Skill 05 机制） |
| 用户成本 | 用户只需一个入口 | 用户需要管理多个对话和交接包 |
| 扩展性 | 增加新能力需要重新设计整体 | 新增 Skill 不影响已有体系 |

### 决策

**采纳六层 Skill 体系**。核心理由：AI 的最大局限是上下文质量随长度降低，分层体系通过隔离上下文来保证每个环节的质量上限。用户管理多个对话的成本，是被标准化交接包机制（Part 7）所补偿的设计取舍。

## ADR-002：手动交接包 vs. 自动函数调用（为何不让 AI 自动传递中间产物）

**状态**：已采纳 · v1.0.0 确立

**决策时间**：接口契约设计阶段

### 背景

实现 Skill 间协作有两种技术方向：
1. **手动交接包**：用户手动复制 YAML 交接包，粘贴到下一个 Skill 的对话
2. **自动函数调用**：通过 Function Calling 或 Agent 框架自动传递上下文

### 分析

| 维度 | 手动交接包 | 自动函数调用 |
|---|---|---|
| 平台依赖性 | 任意平台、任意模型均可使用 | 需要平台支持 Function Calling |
| 用户可见性 | 用户可以看到、编辑、保存交接包 | 过程对用户不透明，不可干预 |
| 错误可调试性 | 交接包内容可视化，出错时可定位到具体字段 | 函数调用失败难以溯源 |
| 实施复杂度 | 极低（只需 YAML 模板） | 需要函数定义、错误处理、重试机制 |
| 用户体验 | 需要手动操作（copy-paste） | 无缝自动化 |
| 成熟度风险 | 无技术风险 | Function Calling 跨模型行为不一致 |

### 决策

**采纳手动交接包机制**。核心理由：体系的首要目标是「能跨任意 AI 平台稳定运行」，手动机制虽然增加用户操作成本，但保证了 100% 的平台兼容性和可调试性。未来如果目标平台稳定支持 Agent 框架，可升级到自动化，但不应将其作为基础依赖。

**后续扩展方向**（非本期范围）：在具备 Function Calling 的平台（如 Dify、Coze）上，可以设计「半自动模式」，交接包由 AI 生成，但需要用户确认后才传递。

## ADR-003：圆桌机制引入（为何在 v1.4.0 引入六角色预执行研讨）

**状态**：已采纳 · v1.4.0 引入

**决策时间**：v1.4.0 设计阶段

### 背景

v1.3.0 体系完成后，观察到以下问题：
- Skill 在执行过程中，因为「只看到自己的视角」，常常输出单一视角的结论
- 没有机制让 Skill 在执行前意识到「我的能力边界在哪里」
- 复杂需求往往需要多个 Skill 协作，但单个 Skill 无法评估协作路径的合理性

### 分析

| 方案 | 描述 | 优缺点 |
|---|---|---|
| 方案 A：不做改变 | 保持现有单 Skill 视角执行 | 简单，但质量上限有限 |
| 方案 B：元提示词增加检查清单 | 每次执行前对照固定问题列表 | 机械，缺乏上下文针对性 |
| 方案 C：六角色圆桌机制 | 六个 Skill 的视角在执行前进行结构化研讨 | 质量高，但增加执行成本 |

### 决策

**采纳圆桌机制（方案 C）**，但设计为**可选触发**而非强制执行（见模式 A/B/C/D/E 的分层设计）。核心理由：复杂需求的预检比执行后返工成本更低；对简单需求则不应增加不必要的仪式感。

圆桌机制的核心设计原则：每个角色的发言不是「我怎么执行」，而是「从我的视角看，这件事有什么风险和盲区」。这种批判性立场是圆桌价值的核心。

## ADR-004：v1.4.1 稳定期建议（为何不持续快速迭代）

**状态**：已采纳 · v1.4.1 记录

**决策时间**：v1.4.1 发布后

### 背景

在 v1.0.0 到 v1.4.1 的迭代过程中，体系快速增加了大量机制（交接包、幻觉防护、飞轮迭代、圆桌机制、双重批判层、记忆锚）。在缺乏足够实战验证的情况下，继续快速增加机制可能带来以下风险：
- 机制之间产生未被发现的耦合
- 用户难以学习和使用不断变化的体系
- 技术债积累（如重复 Part 问题，在 v1.5.0 中被修复）

### 分析

| 维度 | 继续快速迭代 | 进入稳定期 |
|---|---|---|
| 功能丰富度 | 持续增加 | 暂停新增 |
| 稳定性 | 较低 | 较高 |
| 用户学习成本 | 高（规则持续变化）| 低（规则稳定） |
| 缺陷发现 | 被新功能掩盖 | 可以清晰复现和修复 |
| 沙盘推演机会 | 少 | 多 |

### 决策

**采纳稳定期策略**：v1.4.1 之后，以「沙盘推演验证现有机制」为主，以「不引入破坏性变更」为约束，进行小幅优化。v1.5.0 的全量重构（本次）是例外情况，因为发现了结构性 Bug（重复 Part）必须修复。

**稳定期结束条件**：完成至少 3 次完整 Mode C Pipeline 的端到端沙盘推演，验证所有机制协同工作后，才考虑引入下一个重大架构特性。
