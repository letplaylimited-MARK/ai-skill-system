# AI Skill 体系 · 自主迭代飞轮机制
## flywheel-mechanism v1.0.0

---

## 一、飞轮核心逻辑

```
[触发层] ─────────────────────────────────────────────
  │
  ├── 触发1：Skill 05 验收发现 P0/P1 缺陷
  ├── 触发2：用户使用中主动报告问题
  ├── 触发3：定期评估（建议每4周一次）
  └── 触发4：Skill 03 发现更优开源替代方案
  │
  ▼
[分流层] ─────────────────────────────────────────────
  按缺陷类型路由到正确Skill：
  │
  ├── 提示词质量问题（语言/结构/幻觉）   → Skill 01 重新优化
  ├── 工程包结构问题（文件缺失/格式错误）→ Skill 02 阶段04/05
  ├── 开源选型过时（更优方案出现）        → Skill 03 重新评估
  ├── 操作手册错误（步骤/命令/平台适配） → Skill 04 更新手册
  └── 验收标准不足（测试用例覆盖盲区）   → Skill 05（不可自审，上报体系维护者）
  │
  ▼
[修复层] ─────────────────────────────────────────────
  三不原则强制执行（每次修复前检查）：
  │
  ├── 不过度修复：只修改缺陷清单内的问题，不顺带「优化」其他内容
  ├── 不静默修改：所有修改记录在 changelog.md，用户可追溯
  └── 不跳过版本升级：有P0修复→MINOR+1；只P1修复→PATCH+1
  │
  ▼
[版本层] ─────────────────────────────────────────────
  语义化版本号升级（见第三节）
  │
  ▼
[验收层] ─────────────────────────────────────────────
  Skill 05 重新执行受影响的测试用例
  输出验收报告（PASS/CONDITIONAL_PASS/FAIL）
  │
  ▼
[发布层] ─────────────────────────────────────────────
  changelog.md 记录变更
  system-changelog.md 记录体系级里程碑
  GitHub 发布新版本 Tag
  │
  ▼
[反馈层] ─────────────────────────────────────────────
  用户体验改善
  → 下一次触发时缺陷密度降低
  → 飞轮加速
```

## 二、各 Skill 迭代触发条件

### Skill 01 · 超级提示词工程师

| 变更类型 | 版本升级 | 触发条件 |
|---|---|---|
| P0：生成含幻觉的提示词 | MINOR+1 | 验收报告中 hallucination_free=false |
| P0：输出格式规范缺失导致下游Skill无法解析 | MINOR+1 | 接口契约验证失败 |
| P1：特定场景下温度建议不准确 | PATCH+1 | 用户反馈+Skill 05测试 |
| P1：触发词误触发 | PATCH+1 | 触发词冲突测试发现 |
| P2：新增能力（如新语言支持）| MINOR+1 | 用户需求+体系维护者审批 |
| **迭代前提条件** | — | 新版本提示词必须通过Skill 05验收后才可更新SKILL.md |

| 变更类型 | 版本升级 | 触发条件 |
|---|---|---|

| P0：代码块嵌套冲突（三重反引号内嵌套）| MINOR+1 | 验收测试发现 |
| P1：某阶段门控条件描述不清导致跳阶 | PATCH+1 | 用户反馈 |
| P1：生成的安装命令包含不存在的包名 | PATCH+1 | 验收测试发现 |
| P2：新增阶段/子模块 | MINOR+1 | 体系维护者决策 |
| **迭代前提条件** | — | 三不原则检查 + Skill 05重新验收10阶段流程 |

### Skill 03 · 开源技能侦察官

| 变更类型 | 版本升级 | 触发条件 |
|---|---|---|
| P0：虚构了不存在的开源项目 | MINOR+1 | 用户核实后发现 |
| P0：评估矩阵缺少[需用户核实]标注 | MINOR+1 | 防护规则检查失败 |
| P1：某类需求的推导规则覆盖不足 | PATCH+1 | 用户反馈 |
| P2：评估维度权重调整 | MINOR+1 | 体系维护者审批 |
| **定期评估触发** | — | 每4周检查一次重点推荐方案是否仍为最优选 |

### Skill 04 · AI项目执行规划官

| 变更类型 | 版本升级 | 触发条件 |
|---|---|---|
| P0：生成了不适配用户平台的命令 | MINOR+1 | 用户执行失败反馈 |
| P0：断点恢复信息丢失 | MINOR+1 | 验收测试发现 |
| P1：某平台的操作步骤描述过时 | PATCH+1 | 用户反馈 |
| P2：新增平台适配（如Windows子系统）| MINOR+1 | 用户需求 |
| **迭代前提条件** | — | 新版手册必须覆盖原有测试场景 |

### Skill 05 · 测试验收工程师

| 变更类型 | 版本升级 | 触发条件 |
|---|---|---|
| P0：漏判P0缺陷（验收通过但实际有问题）| MINOR+1 | 上线后用户发现问题 |
| P0：自证通过（AI独立判断了需用户手动的测试）| MINOR+1 | 体系维护者审查 |
| P1：测试用例模板覆盖盲区 | PATCH+1 | 新场景发现后补充 |
| **特殊规则** | — | Skill 05自身修复必须由体系维护者（人工）验收，不可自审 |

## 三、版本号升级规则（完整表）

### 单 Skill 版本升级

| 变更类型 | 版本变化 | 示例 |
|---|---|---|
| P0 缺陷修复 | MINOR+1，PATCH归零 | 1.0.2 → 1.1.0 |
| 仅P1 缺陷修复 | PATCH+1 | 1.1.0 → 1.1.1 |
| 新功能（向后兼容）| MINOR+1，PATCH归零 | 1.1.1 → 1.2.0 |
| 破坏性变更（接口/触发词重命名）| MAJOR+1，次版本归零 | 1.2.0 → 2.0.0 |
| 安全修复（紧急）| PATCH+1 + 安全公告 | 1.2.0 → 1.2.1 |
| 仅文档更新 | 无版本升级，changelog记录 | — |

### 体系版本升级（system-changelog.md）

| 变更类型 | 体系版本变化 | 触发条件 |
|---|---|---|
| 任意Skill P0修复 | MINOR+1 | Skill版本升级时 |
| 新增Skill | MINOR+1 | 新Skill发布时 |
| 接口契约破坏性变更 | MAJOR+1 | 接口YAML schema_version变更 |
| 仅P1修复 | 不变 | — |
| 安全修复 | PATCH+1 | — |

## 四、跨 Skill 反馈回路

### 4.1 用户最小操作流程

飞轮的每一次转动，用户的操作负担必须最小化：

```
步骤1：Skill 05 生成「上游反馈包」（FB-001等）
       ↓ 用户操作：确认反馈包内容（30秒）
步骤2：复制反馈包内容
       ↓ 用户操作：Ctrl+C（5秒）
步骤3：打开对应Skill的新对话（如Skill 01）
       ↓ 用户操作：触发词+粘贴（10秒）
步骤4：对应Skill执行修复
       ↓ 用户操作：确认修复方案（1-3分钟）
步骤5：修复完成，触发Skill 02更新工程包（如需要）
       ↓ 用户操作：复制新工程包交接包（10秒）
步骤6：Skill 05重新执行受影响测试
       ↓ 用户操作：手动执行关键测试用例（视情况5-15分钟）
步骤7：验收通过 → 发布新版本
       ↓ 用户操作：确认changelog并发布（2分钟）
```

**设计目标**：一次完整迭代的用户总操作时间 ≤ 30分钟（不含等待AI响应）

### 4.2 缺陷报告标准格式（Skill 05 → 上游Skill）

```yaml
feedback_packet:
  feedback_id: "FB-001"
  to_skill: "skill-01"          # 精准路由
  defect_type: "prompt_quality"
  severity: "P1"
  description: "[具体问题描述]"
  reproduction_steps: "[可复现的操作步骤]"
  expected_behavior: "[期望的正确行为]"
  actual_behavior: "[实际错误行为]"
  suggested_fix: "[建议的修复方向]"
  user_action: "将此反馈包粘贴到Skill 01的对话开头"
```

### 4.3 反馈闭环验证

修复完成后，Skill 05 必须：
1. 重新执行反馈包中对应的测试用例（TC-XXX）
2. 确认 actual_behavior 与 expected_behavior 一致
3. 将该反馈包标记为「已解决」并记录修复版本号
4. 更新 s05-to-release.yaml 中对应的 upstream_feedback_packets 状态

## 五、自举能力（体系用自身改进自身）

| 自举回路 | 触发时机 | 执行过程 | 验收方式 |
|---|---|---|---|
| Skill 01 改进其他Skill提示词 | 任意Skill的提示词质量被标记P0/P1 | 将问题提示词+反馈包送入Skill 01重新优化，输出新版 | Skill 05验收新提示词 |
| Skill 02 重新打包任意Skill | Skill工程包结构P0/P1问题 | 将材料重新送入Skill 02的阶段03，按最新SOP重打包 | Skill 05验收新工程包 |
| Skill 03 发现新工具增强其他Skill | 定期评估（每4周）| Skill 03重新评估相关需求，若发现更优方案，输出「替换建议报告」| 体系维护者决策后执行 |
| Skill 05 验收体系升级 | 每次体系MINOR版本发布后 | Skill 05执行全体系回归测试（所有Skill的核心触发词+接口交接包流转）| 人工最终确认 |

**禁止的自举操作**：
- Skill 05 不可自己验收自己的修复（防利益冲突）
- 自举测试不可替代用户手动执行的关键测试

## 六、飞轮健康度指标

建议追踪以下指标以判断飞轮是否正常运转：

| 指标 | 健康基准 | 告警线 |
|---|---|---|
| 每次迭代用户操作时间 | ≤30分钟 | >60分钟 |
| P0缺陷从发现到修复 | ≤3天 | >7天 |
| 验收通过率（每次迭代）| ≥85% | <70% |
| 连续3次验收都发现新P0 | 正常：0次 | 告警：出现此情况 |
| 接口契约破坏性变更频率 | ≤1次/季度 | >1次/月 |
