# AI Skill 体系 · 沙盘推演框架
## Sandbox Framework v1.0.0

> **核心理念**：不靠堆砌内容让体系变好，靠「主动压测 + 闭环修复」让体系变扎实。
> 沙盘推演是飞轮机制的主动触发层，让你在真实使用出现问题之前，提前发现并修复结构性盲区。

---

## 一、为什么需要沙盘推演

### 1.1 被动飞轮的局限

现有 `flywheel-mechanism.md` 设计的是「被动触发」逻辑：

```
真实使用 → 碰到缺陷 → 用户报告 → 修复 → 再验收
```

这种模式有一个结构性问题：**只能发现用户已经走过的路径上的问题，无法覆盖从未被测试的边界条件**。

### 1.2 沙盘推演的主动价值

```
主动设计场景 → 模拟推演 → 发现盲区 → 预防性修复
```

沙盘推演的三重价值：

| 价值 | 说明 |
|---|---|
| **找漏洞** | 覆盖被动飞轮无法触及的边界场景和异常流 |
| **内化习惯** | 「不翻文档，凭直觉推演」是检验真正掌握程度的唯一方式 |
| **提炼精髓** | 每次推演后提炼 1-2 条规律，积累到最小内核卡 |

### 1.3 三不堆砌原则

沙盘机制本身不应造成系统膨胀：

- **不加新 Skill**：六层已完整，沙盘用于深化而非扩展
- **不加新文档**：修复的产物是让现有文档「更薄且更准确」，不增加新章节
- **不扩大范围**：每次沙盘聚焦一个具体场景，深挖而非铺开

## 二、沙盘的三种模式

| 模式 | 代码 | 用途 | 建议频率 |
|---|---|---|---|
| **全链路沙盘** | SB-FULL | 测试完整六节点流程，找跨 Skill 边界漏洞 | 每2周一次新场景 |
| **单 Skill 单元沙盘** | SB-UNIT | 深挖某一 Skill 的边界条件，设计10个刁钻输入 | 针对已发现问题的 Skill |
| **接口契约沙盘** | SB-CONTRACT | 专门测试交接包在异常情况下的韧性 | 每次接口变更后 |

## 三、单次沙盘标准结构（6阶段）

### Stage 1 · 场景设定（10分钟）

**目标**：明确这次沙盘要测试什么。

```
checklist：
□ 选定一个具体场景（与已测试场景不同）
□ 定义用户角色：
    - 背景：技术水平（小白/中级/专家）
    - 需求模糊程度（1=极清晰 / 5=完全模糊）
    - 跨领域程度（单一领域 / 跨领域混合）
□ 定义「期望体验」：走完六层后用户应该得到什么
□ 标注本次重点测试的 Skill（全链路 or 单 Skill）
```

**输出模板**：

```yaml
sandbox_scenario:
  id: "SB-XXX"
  date: "YYYY-MM-DD"
  mode: "SB-FULL / SB-UNIT / SB-CONTRACT"
  scene_name: "[场景名称]"
  user_profile:
    tech_level: "beginner / intermediate / expert"
    need_clarity: 1-5
    domain_mix: "single / mixed"
  expected_outcome: "[走完六层后的期望产出]"
  focus_skill: "all / skill-XX"
```

---

### Stage 2 · 冷推演（20-30分钟）

**目标**：在不翻任何文档的情况下，凭直觉走一遍六节点流程。

**规则**：
- ❌ 不翻 USER-GUIDE.md
- ❌ 不看激活卡（activation-card.md）
- ❌ 不看 SUPER-SYSTEM-PROMPT
- ✅ 只凭脑中的认知推演每一步

**每个节点回答三个问题**：

```
节点：[Skill 编号 · 名称]
Q1: 用户此刻需要做什么？（具体操作）
Q2: AI 应该输出什么？（期望输出格式/内容）
Q3: 交接包里应该包含哪些字段？
```

**卡顿记录格式**：

```
卡顿点 #N
位置：[Skill XX · 节点名称]
问题：[具体卡在了什么地方]
类型：□ 操作步骤不清楚  □ 输出格式不确定  □ 交接包字段模糊  □ 不知道该去哪个 Skill
严重度：□ 完全卡住  □ 模糊但能继续  □ 轻微疑惑
```

---

### Stage 3 · 热推演（30-60分钟）

**目标**：拿出 SUPER-SYSTEM-PROMPT 实际走一遍六节点，记录真实体验与期望体验的 GAP。

**操作流程**：
1. 激活 SUPER-SYSTEM-PROMPT（粘贴到新对话）
2. 输入 Stage 1 设定的场景描述
3. 按六节点顺序走一遍，每个节点记录：

```
节点评估 · [Skill XX]
输入：[实际输入内容摘要]
输出质量：[1-5 分] + 具体评价
交接包质量：[完整 / 缺字段 / 格式错误]
用户操作摩擦：[操作步骤是否清晰 / 有无歧义]
与冷推演的差异：[实际结果 vs Stage 2 的预期]
```

**GAP 汇总**：

```
GAP-001：[冷推演预期] vs [热推演实际]
GAP-002：...
```

---

### Stage 4 · 缺陷归档（10-15分钟）

**目标**：将发现的问题系统化，按飞轮机制路由。

**缺陷分级**：

| 级别 | 定义 | 处理速度 |
|---|---|---|
| P0 | 导致流程完全中断 / 输出严重错误 | 立即修复 |
| P1 | 体验明显受损，用户会感到困惑 | 下次迭代修复 |
| P2 | 轻微不完善，不影响主流程 | 积压后集中处理 |

**缺陷报告格式（引用自 flywheel-mechanism.md）**：

```yaml
feedback_packet:
  feedback_id: "SB-XXX-FB-001"
  source: "sandbox"          # 来源标记为沙盘
  to_skill: "skill-XX"
  defect_type: "[prompt_quality / interface_contract / ux_friction / rule_gap]"
  severity: "P0 / P1 / P2"
  description: "[具体问题描述]"
  reproduction_steps: "[可复现的操作步骤]"
  expected_behavior: "[期望的正确行为]"
  actual_behavior: "[实际错误行为]"
  suggested_fix: "[建议的修复方向]"
```

**路由规则（遵循 flywheel-mechanism.md §二）**：

```
提示词质量问题    → Skill 01
工程包结构问题    → Skill 02
开源选型过时      → Skill 03
操作手册错误      → Skill 04
验收标准不足      → Skill 05（上报体系维护者，不可自审）
接口契约问题      → 更新 interface-contracts/*.yaml
体系级流程问题    → 更新 SUPER-SYSTEM-PROMPT.md / USER-GUIDE.md
```

---

### Stage 5 · 精准修复（时间视缺陷数量而定）

**目标**：按缺陷报告执行最小化修复。

**三不原则**（强制遵守）：

```
□ 不过度修复：只修改缺陷清单内的问题，不顺带「优化」其他内容
□ 不静默修改：所有修改记录在对应 Skill 的 changelog.md
□ 不跳版本：P0 修复 → MINOR+1；仅 P1 修复 → PATCH+1
```

**修复完成确认清单**：

```
□ 修改了具体文件：[文件路径]
□ 更新了 changelog（Skill 级别）
□ 对应缺陷报告标记为「已修复」
□ 版本号正确更新
```

---

### Stage 6 · 最小内核提炼（10分钟）

**目标**：从本次沙盘提炼 1-2 条「无需文档即可记住」的规律，追加到 `minimum-viable-card.md`。

**提炼模板**：

```
本次沙盘洞察：
洞察 #N：[一句话描述这条规律]
来源：[来自哪个卡顿点 / GAP / 缺陷]
记忆钩子：[让这条规律难以忘记的比喻或类比]
```

## 四、沙盘记录文件规范

### 4.1 文件命名

```
/workspace/ai-skill-system/system/sandbox-logs/
  ├── sandbox-log-001.md    # 第一次全链路沙盘
  ├── sandbox-log-002.md    # 第二次沙盘（场景不同）
  └── sandbox-index.md      # 沙盘汇总索引
```

### 4.2 文件结构模板

每个沙盘日志文件包含：

```markdown
# 沙盘日志 #XXX · [场景名称]
## 元数据（Stage 1 输出）
## 冷推演记录（Stage 2 · 卡顿清单）
## 热推演记录（Stage 3 · GAP 清单）
## 缺陷归档（Stage 4 · feedback_packets）
## 修复记录（Stage 5 · 变更摘要）
## 本次洞察（Stage 6 · 精髓提炼）
```

## 五、沙盘健康度指标

建议在 `sandbox-index.md` 中追踪：

| 指标 | 健康基准 | 告警线 |
|---|---|---|
| 冷推演卡顿点数量（趋势）| 逐次减少 | 连续3次不下降 |
| 每次沙盘发现的 P0 缺陷数 | ≤1 | ≥3 |
| 从沙盘发现到修复完成时间 | ≤2天（P1）| >7天 |
| 精髓卡词条增长（来自沙盘）| 每次 1-2 条 | 0条（推演收获为零） |
| 冷热推演一致性（GAP 数量）| ≤3个 GAP | ≥8个 GAP（内化不足）|

## 六、与被动飞轮的协同关系

```
主动层（沙盘推演）      被动层（飞轮）
      │                      │
  触发0：主动压测        触发1-4：真实使用
      │                      │
      └──────┬───────────────┘
             │
        [统一缺陷归档]
        feedback_packet
             │
        [分流修复层]
        按 Skill 路由
             │
        [版本发布层]
        changelog + tag
```

两层机制互补：
- **沙盘主动触发**：覆盖「从未走过的路径」
- **飞轮被动触发**：覆盖「真实使用中发现的问题」
- **统一缺陷格式**：所有问题无论来源，都通过 feedback_packet 标准化归档

---

## 七、快速启动指南

第一次运行沙盘，按此顺序：

```
1. 复制下方场景设定模板，填写你的场景
2. 不翻任何文档，先做「冷推演」（Stage 2）
3. 再用 SUPER-SYSTEM-PROMPT 实际跑一遍（Stage 3）
4. 对比两次，填写 GAP 清单
5. 有 P0/P1 问题？→ 填 feedback_packet，按路由修复
6. 提炼本次洞察 → 追加到 minimum-viable-card.md
7. 更新 sandbox-index.md 添加本次记录条目
```
