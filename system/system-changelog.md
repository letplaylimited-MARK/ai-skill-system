## System v1.5.1 — 2025年（当前版本）

### 飞轮建议回归 · FW-001 / FW-002 修复落地

本次为 v1.5.0 的补丁版本，将集成测试报告（v1.2.0）中记录的两项飞轮建议正式回归至 Skill 04 提示词。这是飞轮机制首次完整闭环：建议写入 → 版本积压 → 本次修复。

| 变更文档 | 说明 |
|---|---|
| `skill-04-planner/core/system-prompt-full.md`（v1.0.0 → v1.1.0）| **FW-001 回归**：Step 模板新增必填 `error_handling` 字段；**FW-002 回归**：新增执行原则 E8（配置文件模板强制检查）；新增「配置文件强制检查」节；激活卡版本号同步至 v1.1.0 |
| `skill-04-planner/core/activation-card.md`（版本同步）| 版本号更新至 v1.1.0；新增 FW-001/FW-002 变更说明 |
| `ai-skill-web-app/src/lib/skill-prompts/s04.txt`（同步更新）| Web 平台 Skill 04 提示词同步回归 FW-001/FW-002 内容，版本升至 v1.1.0 |
| `system/system-changelog.md`（本文档，v1.1.0 → v1.2.0）| 新增本条目；修正历史条目中 system-changelog 本身的版本号 |

**FW-001 详情（Skill 04 Step 模板 error_handling 字段）**：
- 来源：v1.2.0 端到端集成测试报告飞轮建议
- 问题：集成测试发现 Skill 04 生成的操作手册中，步骤失败时无明确处理指引，用户需自行判断是否继续/回滚
- 修复：Step 模板强制新增 `error_handling` 字段，值为可执行的失败处理方案（重试/回滚/跳过/联系维护者）
- 影响范围：Skill 04 v1.0.0 → v1.1.0，所有 Step 输出格式变更（向后兼容，旧交接包无需重新生成）

**FW-002 详情（配置文件模板强制检查）**：
- 来源：v1.2.0 端到端集成测试报告飞轮建议
- 问题：集成测试中 Skill 04 生成「创建 .env 文件」步骤时未附配置模板，用户不知道需要填写哪些字段
- 修复：新增执行原则 E8，凡涉及配置文件的 Step 必须附上含注释的配置模板；SOP 未提供时追问用户
- 影响范围：Skill 04 v1.0.0 → v1.1.0，手册生成阶段（Stage 03）行为变更

**关键学习**：
- **L-010**：飞轮建议的价值在于被执行，而非被记录。FW-001/FW-002 从 v1.2.0 集成测试报告写入到本次修复，跨越了约 3 个版本周期（v1.2.0 → v1.3.0 → v1.4.0 → v1.4.1 → v1.5.0 → v1.5.1）。**飞轮建议应在下一个版本就执行，而非积压**。本体系现在将此作为发布门控：每次 MINOR/PATCH 发布前，必须检查是否有积压的飞轮建议。
- **L-011**：「配置文件模板缺失」是 AI 生成操作手册时最常见的实用性缺陷。用户能找到配置文件的位置，但不知道里面填什么。强制要求模板输出将显著提高操作手册的可执行率。

**Web 平台同步说明**：
- `ai-skill-web-app/src/lib/skill-prompts/s04.txt` 同步更新至 v1.1.0 等效版本
- 两套提示词（工程包完整版 + Web 平台精简版）现已重新对齐
- 建议后续建立脚本自动同步机制（见 ROADMAP.md TD-03）

**结构性 Bug 修复详情（P0）**：
- 旧 Part 7（HP-A=S01→S02）与新 Part 7·（交接包A=S00→Any）命名冲突 → 统一为 HP-D/A/B/C/E/F
- 旧 Part 10（速查卡，缺 S01/S02）与新 Part 10·（重复）重叠 → 合并为单一完整版
- Part 2 激活确认输出中残留旧版本号 v1.2.0 → 更新为 v1.5.0

**关键学习**：
- **L-007**：技术债比功能缺失更难发现。重复 Part 问题在 v1.2.0 就已存在，持续到 v1.4.1（约 3 个版本周期）才被识别。原因是每次新增内容时直接追加，没有对全文结构做验证。**结构性检查应作为每次发布的前置门控**。
- **L-008**：命名系统是架构的第一等公民。HP-A 在两个不同的 Part 中指代不同含义（S01→S02 vs S00→Any），这种歧义会在用户实际使用时造成严重混淆。**每个新名称引入时必须检查全文是否有同名但含义不同的标识符**。
- **L-009**：原子化重构的价值在于「迫使发现隐性耦合」。本次全量重构过程中，发现了 S01/S02 缺少 council-role.md（所有其他 Skill 都有）、interface-contracts/ 目录缺少索引文档等 6 处不一致。这些问题在增量迭代中容易被忽视，但在全量重构时会被系统性地暴露。

## System v1.4.1 — 2025年（当前版本）

### 修订：圆桌记忆锚 · 补足共识连续性

本次为 v1.4.0 的小修订版本，针对「圆桌共识在执行链路中断裂」的结构性缺口，新增圆桌记忆锚机制。

| 变更文档 | 说明 |
|---|---|
| `SUPER-SYSTEM-PROMPT.md` Part 11 §11.5（更新）| 圆桌报告末尾新增「一行决策卡」：三个选项（开始/调整/提问）+ P0 风险自动门控提示 |
| `SUPER-SYSTEM-PROMPT.md` Part 13（新增）| 圆桌记忆锚机制：四个组件（约定回显/风险高亮/前序自审接收/P0门控）、轻量原则（无内容不显示）、EG-09/EG-10 防错护栏 |
| `minimum-viable-card.md`（更新）| 「完整流程」图加入圆桌层与记忆锚标注，反映真实体系架构 |

**设计平衡点**：
- 记忆锚只提取与「当前 Skill」相关的内容，不重复圆桌全文（轻量）
- P0 门控需用户确认，其余组件静默展示不增加决策负担（非侵入）
- 无相关内容时整个组件不显示（不产生信息噪音）

**关键学习**：
- **L-001**：机制完整性 ≠ 机制有效性。记忆锚机制在设计时看起来完整，但在实际执行中「圆桌共识如何被下游 Skill 接收」这一环节仍是假设，尚未经过真实沙盘验证。
- **L-002**：「稳定期」的价值在于让现有机制被充分使用和压测，而不是继续堆砌。v1.4.1 是第一次明确提出稳定期建议，后续应以此为原则。

**战略建议**：v1.4.1 之后建议进入稳定期，下一个 MINOR 版本（v1.5.0）应由真实沙盘/使用中发现的缺陷驱动，而非架构设想驱动。

**设计原则**：
- 圆桌自动触发，无需用户显式调用（说「跳过圆桌」才豁免）
- 六角色全员发言，`capability_cannot` 字段强制填写（防空洞批判）
- 交接包向后兼容（v1.0 仍有效，v1.1 可选字段鼓励使用）

**关键学习**：
- **L-003**：「六角色圆桌」的核心价值不在于增加角色数量，而在于「强制声明能力边界」——每个角色必须说「我做不了什么」。这种结构性诚实是防止 AI 过度自信的最有效机制之一。
- **L-004**：圆桌机制设计为可选触发而非强制，是正确的权衡。强制圆桌会让简单任务产生不必要的仪式感，削弱用户体验。五种模式（A/B/C/D/E）的分层设计（v1.5.0 正式化）是这一认知的延伸。

## System v1.3.0 — 2025年

### 里程碑：引入沙盘推演框架 · 巩固而非堆砌

本次更新基于核心设计洞察「不靠堆砌内容，靠主动压测让体系扎实」，引入沙盘推演机制作为飞轮的主动触发层。

| 新增/变更文档 | 说明 |
|---|---|

| `minimum-viable-card.md`（新建）| 体系精髓单页卡，3条核心规则 + 六角色速查 + 60秒流程图，目标：不翻其他文档就能使用体系，随每次沙盘持续精炼 |
| `system/flywheel-mechanism.md`（v1.0.0→v1.1.0）| 在触发层新增「触发0：主动沙盘压测」，新增第七节主被动对比，与 sandbox-framework.md 建立互引关系 |

**设计原则**（三不堆砌）：
- 不加新 Skill（六层已完整）
- 不加新文档（让现有文档「更薄且更准确」）
- 不扩大范围（每次沙盘深挖一个具体场景）

**关键学习**：
- **L-005**：「巩固」比「扩展」更难。在体系功能完整的情况下，最大的挑战是抵制「再加一个功能」的诱惑，专注于让现有功能稳定可靠地运作。沙盘推演机制正是为此设计的反馈闭环。
- **L-006**：minimum-viable-card 的价值在于它是「体系的可执行摘要」。当用户不需要翻阅完整文档就能开始使用体系时，说明体系的复杂度被成功封装了。这是衡量体系成熟度的一个实用指标。
