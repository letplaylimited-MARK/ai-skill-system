## 一、体系总览

### 1.1 设计哲学

本体系基于三个核心认知进行设计：

**认知一：Skill 是系统提示词，不是函数**
每个 Skill 本质上是一段定义 AI 角色的系统提示词。Skill 间的「协作」不是程序调用，而是用户在不同 Skill 之间搬运标准化的中间产物（交接包）。体系的核心设计目标是：让每次搬运尽可能简单、无损、标准化。

**认知二：六个 Skill 独立存在，相互正交**
每个 Skill 解决一类专属问题，职责边界清晰，不相互包含。用户可以单独使用任意一个 Skill，也可以按需组合使用。

**认知三：体系通过用户驱动的飞轮自主进化**
体系的迭代不依赖自动化，而是通过 Skill 05（测试验收）发现问题、标准化缺陷报告、路由到对应 Skill 修复的闭环实现持续改善。

### 1.3 六个 Skill 精确职责边界

| 编号 | 名称 | 一句话职责 | 输入 | 输出 | 不做什么 |
|---|---|---|---|---|---|
| **Skill 00** | 智能体导航官 | 接收任意需求 → 意图识别 → 推荐使用哪个Skill | 用户模糊需求/任意描述 | 路由推荐+交接包A | 不执行任何具体任务 |
| **Skill 01** | 超级提示词工程师 | 原始提示词/材料 → 工程化结构化系统提示词 | 原始提示词或角色描述 | 优化提示词三档+交接包C | 不判断是否需要提示词，不搜索开源方案 |
| **Skill 02** | SOP 工程师 | 需求+提示词 → 完整Skill工程包（10阶段）| 需求描述+可选交接包 | 完整工程包+交接包D | 不优化提示词语言质量，不搜索开源，不实际部署 |
| **Skill 03** | 开源技能侦察官 | 项目需求 → 开源搜索+七维评估+选型报告 | 项目需求描述 | 评估报告+交接包B | 不开发新Skill，不优化提示词，不做最终架构决策 |
| **Skill 04** | AI项目执行规划官 | 项目需求 → 人工可执行的多Skill协作操作手册 | 项目需求+可选交接包 | 操作手册（含断点恢复）| 不自动执行任何步骤，不替代其他Skill功能 |
| **Skill 05** | 测试验收工程师 | Skill工程包 → 测试+验收报告+上线决策 | Skill工程包/交接包D | 验收报告+上游反馈包 | 不独立验收自身修复（防利益冲突），关键测试必须用户手动执行 |

## 二、接口契约体系

### 2.1 交接包标准结构

所有 Skill 的输出必须包含标准化交接包（handoff payload），格式如下：

```yaml
handoff:
  schema_version: "1.1"
  from_skill: "skill-XX"
  to_skill: "skill-YY"
  payload:
    [Skill特有字段]
  user_action: "将此交接包复制，粘贴到Skill YY的对话开头"
  created_at: "2025-XX-XX"
  self_review:          # 可选，v1.1新增
    assumptions: []
    potential_failures: []
    predicted_deduction_by_s05: ""
  downstream_notes:     # 可选，v1.1新增
    to_skill: ""
    cautions: []
    required_verification: []
```

### 2.2 六个接口契约文件清单

| 文件 | 来源 | 目标 | 核心传递内容 |
|---|---|---|---|
| `s00-to-any.yaml` | Skill 00 | 任意Skill | 意图类型/置信度/路由推荐/路由原因 |
| `s03-to-s02.yaml` | Skill 03 | Skill 02 | 评估矩阵/选定方案/缺口列表/打包请求 |
| `s01-to-s02.yaml` | Skill 01 | Skill 02 | 三档优化提示词/质量评分/温度建议/环境假设 |
| `s02-to-s05.yaml` | Skill 02 | Skill 05 | 工程包元数据/测试目标/已知限制/缺陷历史 |
| `s04-to-user.yaml` | Skill 04 | 用户 | 环境信息/前提条件/分步操作/断点恢复 |
| `s05-to-release.yaml` | Skill 05 | 发布决策 | 验收结论/测试结果/风险评估/上游反馈包 |

详细规范见 `/interface-contracts/` 目录中各 YAML 文件。

## 三、幻觉防护体系

### 3.1 全局幻觉防护规则

以下规则适用于所有六个 Skill，无一例外：

| 规则编号 | 规则内容 | 违反后果 |
|---|---|---|
| G-HAL-01 | 实时数据（Star数/更新日期/版本号）必须标注「[需用户核实]」| P0缺陷 |
| G-HAL-02 | 代码/命令示例必须附注「执行前请验证包名和版本」| P1缺陷 |
| G-HAL-03 | 不虚构不存在的工具/库/功能/参数 | P0缺陷 |
| G-HAL-04 | 部署步骤前必须收集目标环境信息（OS/版本/权限）| P1缺陷 |
| G-HAL-05 | Skill 05不可独立验收自身同类产出（须用户手动执行关键测试）| P0缺陷 |

### 3.2 Skill 专属幻觉风险

| Skill | 高风险点 | 防护机制 |
|---|---|---|
| Skill 03 | 虚构开源项目的Star数/活跃度 | 评估矩阵模板内嵌[需用户核实]字段 |
| Skill 02 | 生成不可运行的安装命令 | 每个代码块后强制附注验证提示 |
| Skill 04 | 生成不适配用户平台的部署步骤 | 第一步必须完成环境信息收集表 |
| Skill 01 | 优化提示词时引入不存在的工具能力 | 接口中必须传递环境约束清单 |
| Skill 05 | 自证通过（AI验收AI产出）| 关键测试步骤明确标注「必须用户手动执行」|

## 四、自主迭代飞轮机制

### 4.1 飞轮运转逻辑

```
[触发层] 任何问题信号
  │
  ├── Skill 05 验收发现 P0/P1 缺陷
  ├── 用户在使用中报告问题
  └── 定期评估（建议每4周一次）
  │
  ▼
[分流层] 按缺陷类型路由
  │
  ├── 提示词质量问题    → Skill 01 重新优化
  ├── 工程包结构问题    → Skill 02 阶段04/05
  ├── 开源选型过时      → Skill 03 重新评估
  ├── 操作手册错误      → Skill 04 更新手册
  └── 验收标准不足      → Skill 05（不可自审，需上报体系维护者）
  │
  ▼
[修复层] 三不原则强制执行
  不过度修复 / 不静默修改 / 不跳过版本升级
  │
  ▼
[版本层] 语义化版本升级
  P0修复 → MINOR+1 | P1修复 → PATCH+1 | 破坏性接口变更 → MAJOR+1
  │
  ▼
[验收层] Skill 05重新验收（修复后必须）
  │
  ▼
[发布层] system-changelog.md 记录
  │
  ▼
[反馈层] 用户体验改善 → 飞轮加速
```

### 4.2 版本升级规则表

| 变更类型 | 单Skill版本 | 体系版本 |
|---|---|---|
| P0 缺陷修复 | MINOR+1（如1.0→1.1）| MINOR+1 |
| P1 缺陷修复 | PATCH+1（如1.1.0→1.1.1）| 不变 |
| P2 新功能 | MINOR+1 | 不变 |
| 接口契约破坏性变更 | MAJOR+1 | MAJOR+1 |
| 新增 Skill | 不影响已有 | MINOR+1 |
| 安全修复（紧急）| PATCH+1 + 安全公告 | PATCH+1 |

### 4.3 自举能力（体系用自身改进自身）

| 自举回路 | 操作 |
|---|---|
| Skill 01 改进其他Skill的系统提示词 | 将其他Skill的提示词送入Skill 01优化，产出新版本 |
| Skill 02 重新打包任意Skill | 将任意Skill的材料送入Skill 02重新工程化 |
| Skill 03 发现新工具增强其他Skill | 侦察结果触发Skill 02更新对应Skill的工具推荐 |
| Skill 05 验收体系自身升级 | 每次体系版本升级后，Skill 05执行全体系回归测试 |
